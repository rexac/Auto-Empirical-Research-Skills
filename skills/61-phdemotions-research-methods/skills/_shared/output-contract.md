# Output Contract — Research Audit Reports

> Every research audit skill produces output in **exactly** this shape.
> Identical structure across all audits enables diffing, comparison, and progress tracking.

## File location

Reports are saved to:
```
<project-root>/docs/audits/<skill-slug>-<YYYY-MM-DD>.md
```

Create `docs/audits/` if it does not exist. If a report already exists for today, append a numeric suffix (`-2`, `-3`, ...).

Also print the report inline so the researcher reads it without opening the file.

## Report template

```markdown
# Research Audit — <Project Name>
## Lens: <Audit Skill Name>
**Date:** <YYYY-MM-DD>
**Auditor:** Claude (skill: <skill-slug>)
**Scope:** <one sentence — what was audited and what was not>
**Materials reviewed:** <list of files/data/docs inspected>

---

## Verdict

**<Ready to submit / Needs revision / Major issues / Not ready>**

---

<repeat>
```

## Rules

1. **No meta-hedging.** Write as the reviewer persona would: with authority. No "I noticed" or "it seems."
2. **Every finding cites evidence.** File paths, line numbers, variable names, statistical output. If you cannot cite it, cut it.
3. **"Why it matters" must be persona-specific.** "This isn't ideal" is not a reason. "A Reviewer 2 at JCR would flag this as insufficient reporting and require a major revision" is.
4. **The verdict is holistic.** 0 Blockers + 12 Majors can still be "Needs revision." Use judgment.
5. **The reviewer-voice paragraph is the most important line.** Screenshot-worthy, specific, true to the persona.
6. **Wins are mandatory.** Every audit must note at least one thing done well, if anything positive exists.
